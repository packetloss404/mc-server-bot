WHAT'S ACTUALLY WORKING

Worker Thread Isolation Gets It Right

Each bot runs in its own worker thread with IPC message passing and 60-second timeouts. One bot crashing doesn't cascade to the fleet. The WorkerHandle implements crash recovery with up to 5 retries.

The IPCChannel uses an RPC-style pattern with pending request tracking that prevents deadlocks. This is genuinely well-engineered and the right architectural choice for multi-bot orchestration.

APPROVED

Voyager Learning Pipeline Has Real Bones

The CurriculumAgent → ActionAgent → CriticAgent → SkillLibrary pipeline mirrors the original Voyager research paper's architecture. Tasks are proposed, code is generated, execution is evaluated, and successful skills are stored for reuse.

The pipeline supports task decomposition, quality scoring, and automatic difficulty progression. The concept is sound even though the generated skills are currently broken.

APPROVED

Fleet Control Is Impressively Scoped

CommandCenter, MissionManager, SquadManager, RoleManager, MarkerStore — this is a serious command-and-control framework. Missions have lifecycle management (create, start, pause, cancel, retry). Squads have membership CRUD. Roles enforce one-per-bot with autonomy levels.

The social memory system with decay, emotional states, and reflection is creative and well-designed, even if underutilized by current behavior code.

APPROVED

SECURITY — STOP EVERYTHING AND FIX THESE

SEC-001CRITICAL

Hardcoded Google API Key

A live GOOGLE_API_KEY sits in .env checked into the repo. Revoke it today. Scrub from git history with BFG. This is a 15-minute fix with existential consequences if ignored.

.env:1
SEC-002CRITICAL

Path Traversal

The skill-reading endpoint joins user input directly into a file path. ../../../etc/passwd reads arbitrary files off the server. Use path.resolve() and validate containment.

src/server/api.ts:331
SEC-003CRITICAL

Zero Authentication

All 30+ REST endpoints and WebSocket connections are completely open. No API key, no JWT, no sessions. Anyone on the network can spawn bots, delete bots, and read all data.

src/server/api.ts (all)
SEC-004CRITICAL

No Rate Limiting

An attacker can spawn unlimited bots, queue thousands of tasks, or burn through LLM API quota. Add express-rate-limit immediately.

src/server/api.ts
SEC-005HIGH

API Key in URL Params

Gemini client passes key= in the URL query string. Shows in logs, memory dumps, and heap snapshots saved to diagnostics/. Use Authorization header.

src/ai/GeminiClient.ts:23
SEC-006HIGH

No Input Validation

Bot names, chat messages, task descriptions, and coordinates pass through without sanitization, length limits, or bounds checking. XSS via dashboard, coordinate DoS, name injection all possible.

CODE QUALITY — THE ENGINEERING AUDIT

The any Epidemic: 157+ Violations

The single biggest engineering debt in the codebase. Mineflayer's bot object is cast to any 20+ times in BotInstance alone. Worker IPC proxies use as any to bridge type mismatches. LLM responses are parsed into any with zero schema validation.

The effect is a TypeScript codebase that provides almost no compile-time safety on its most critical paths. Typos in property names, async/sync mismatches in proxies, and structural changes in API responses will all surface as runtime errors — in production, with no test safety net.

THE SKILLS LIBRARY — A GRAVEYARD OF HALLUCINATIONS

Generated Skills Call Functions That Don't Exist

The Voyager pipeline generates skill code via LLM, then stores it in skills/*.js. The problem: the LLM hallucinates function names. Multiple skills reference mineBlock(), craftItem(), and placeItem() — functions that do not exist in the action library.

// skills/craft_a_stone_pickaxe.js
async function craftAStonePickaxe(bot) {
  await mineBlock("cobblestone", 3);  // DOESN'T EXIST
  await craftItem("stone_pickaxe");   // DOESN'T EXIST
  await placeItem("crafting_table");  // DOESN'T EXIST
}

Every skill in skills/index.json shows successCount: 0 and failureCount: 1 with optimistic "quality" scores of 0.6-0.7. The critic agent isn't catching these failures.

Additionally, there are dozens of near-duplicates: craft_a_stone_pickaxe.js, craft_1_stone_pickaxe.js, craft_1_stone_pickaxe_using.js — no de-duplication.

CONDEMNED

Personalities Are a Beautiful Lie

Six personality types — merchant, guard, explorer, farmer, blacksmith, elder — each with evocative descriptions. The merchant "loves trading, haggling, and rare goods." The guard is "protective, suspicious of strangers." The farmer is a "cheerful crop tender."

In reality, every personality uses identical action libraries, identical task selection, identical inventory management. The only difference is a systemPromptFragment that changes the chat style.

Root cause: PersonalityType.ts only modifies the chat system prompt. Personality should drive task selection priority, behavioral triggers, and specialized action sets.

COSMETIC ONLY

THE AI BRAIN — FUNCTIONAL BUT FRAGILE

What Works

Clean LLMClient interface abstracts Anthropic and Gemini. Adding a provider means implementing one interface. System prompts are well-structured with context, inventory, and blocker state. The dual-provider architecture is genuinely good.

SOLID

No Retry, No Backoff

A transient 429 or 503 from the LLM API becomes a permanent task failure. Hard 60-second timeout with zero retry. No exponential backoff. No circuit breaker for cascading failures. Multiple bots can flood the API simultaneously with no queue or coordination.

FRAGILE

No Output Validation

LLM-generated code is executed without AST validation against the known action API. If the LLM hallucinates function names (it does — see skills), broken code enters the skill library permanently. Thinking responses from Gemini are silently discarded at GeminiClient.ts:151.

DEPLOYMENT & OPERATIONS

No Container, No CI

No Dockerfile. No docker-compose. No CI/CD pipeline. Deployment is node dist/index.js > /tmp/dyobot.log & disown. The watchdog is nice but this isn't production infrastructure.

Flat-File Everything

All persistence is JSON files in data/. Atomic writes via temp file + rename (good). But no transaction safety across files, no read-before-write locking, unbounded growth with no TTL cleanup. Fine for <20 bots. Will break at scale.

Socket.IO Flooding

Every 10 seconds, all bots emit position/health/state to all connected clients individually. With 50 bots and 20 clients: 50,000 messages/minute. Should batch into single bots:batch-update event.
