DyoBot has a genuinely impressive Voyager AI pipeline — real LLM-driven autonomous code generation, a skill library with hybrid TF-IDF/embedding search, and a 17-page dashboard with polished UI. WHAT GENUINELY WORKS

The Voyager Pipeline Is Real and Impressive

The CurriculumAgent → ActionAgent → CriticAgent → SkillLibrary pipeline is not a stub. It actually calls the LLM, generates executable code, runs it through a Babel AST parser, retries up to 3 times with feedback from the critic, and stores successful skills for reuse.

Code generation at ActionAgent.ts uses structured prompts with inventory context, personality fragments, nearby blocks, and blocker state. The 3-retry feedback loop with critic evaluation is a genuinely sophisticated design that mirrors the Voyager research paper architecture.

REAL & WORKING

Skill Library: Hybrid Search with Quality Gating

The SkillLibrary implements TF-IDF plus embedding-based hybrid search for skill retrieval. Quality scoring gates which skills get reused. The concept of learning from successful execution and building a reusable library of behaviors is well-implemented.

Personality-driven task pools are real: CurriculumAgent.ts:372 gives a 65% chance to select personality-specific tasks first. Guards get different task proposals than farmers. This is genuine behavioral differentiation at the curriculum level.

REAL & WORKING

Dashboard UI Is Polished and Extensive

17 pages across manage, fleet, map, chat, activity, stats, skills, build, chains, roles, social, commander, coordination, routines, metrics, and history. Loading states, error boundaries, and responsive design throughout. The SocketProvider correctly subscribes to backend events.

Socket infrastructure exists on both sides — the backend emits events for commands, missions, markers, zones, squads, roles, and bot state. The frontend has Zustand stores ready to consume them. The plumbing is 90% there; it just writes to the wrong stores.

REAL & WORKING

“The Voyager pipeline is the real thing — LLM-driven task proposal, code generation with Babel AST parsing, critic evaluation, and skill library storage. This is genuinely sophisticated AI agent architecture. Everything else is a beautiful shell waiting to be connected to it.” — Audit Panel, AI Infrastructure Division

THEME 1 — THE POTEMKIN VILLAGE

Five Control Managers: Fully Built, Never Instantiated

The control platform — CommandCenter, MissionManager, MarkerStore, SquadManager, RoleManager — represents thousands of lines of carefully engineered code. Each manager has full lifecycle management, persistence, and Socket.IO event emission.

But grep the entire src/ tree for new CommandCenter, new MissionManager, new MarkerStore, new SquadManager, or new RoleManager and they appear only in test files under test/control/. Zero instantiations in production code. The api.ts route handlers reference a commanderService that IS instantiated (line 67), but the five underlying managers it would delegate to are not.

Similarly, BuildCoordinator and ChainCoordinator are never instantiated outside their own files. The supply chain coordinator at src/supplychain/ChainCoordinator.ts writes to disk but is never constructed by any calling code.

// What exists in test/control/ ONLY:
new CommandCenter(bm, io);     // integration.test.ts:64
new MissionManager(bm, io);    // integration.test.ts:65
new MarkerStore(io);           // integration.test.ts:66
new SquadManager(io);          // integration.test.ts:67

// What exists in production: nothing.

~35 of 50 Frontend API Calls Hit 404

The dashboard's 17 pages make approximately 50 distinct API calls. About 35 of these target endpoints that either do not exist or hit routes that delegate to the never-instantiated managers. CLAUDE.md documents all of these endpoints as if they work. The API section describes POST/GET endpoints for commands, missions, markers, zones, routes, squads, roles, and commander operations. Roughly 80% of this documentation describes fictional functionality.

THEME 2 — THE AI PIPELINE

Core Pipeline: Genuine LLM Integration

The Voyager loop at VoyagerLoop.ts orchestrates real LLM calls through a clean LLMClient interface. The CurriculumAgent proposes tasks using world state, inventory, and personality context. ActionAgent generates executable JavaScript. CriticAgent evaluates results. Successful code becomes reusable skills.

The prompt engineering is well-structured: personality fragments from src/ai/prompts/personality.ts, action primitive documentation, inventory state, and blocker history all compose into effective LLM context. This is production-quality prompt architecture.

SOLID

CommanderService: Regex Pretending to Be AI

CommanderService accepts an LLMClient in its constructor (CommanderService.ts:88) but never calls it. The parse() method at line 387 uses a chain of /\b(pause|stop|halt)\b/.test() regex matches to derive intent and confidence scores. No LLM is involved despite the interface suggesting otherwise.

Worse, execute() at line 530 returns empty arrays immediately after tracking metrics on the empty result. It creates a CommanderExecuteResult with { commands: [], missions: [] } and records it as "executed." This is a no-op that logs success.

// CommanderService.ts:540-551
const result: CommanderExecuteResult = {
  commands: [], missions: []
};
this.trackExecutionMetrics(result);
// ... records as status: 'executed'
return result;

STUB

Broken Sentiment & Missing Rate Limiting

The sentiment analysis in src/ai/prompts/chat.ts:2 declares const NEGATIVE_WORDS: string[] = [] — an empty array. The scoring function at line 7 filters against this empty list, so negative sentiment is always zero. Affinity adjustments based on player chat tone are impossible.

The config option maxConcurrentRequests: 3 exists in config.yml:61 and is typed in config.ts:56, but is never referenced anywhere in the LLM client code. Multiple bots can flood the API simultaneously with zero coordination or queuing.

The codebase supports only a single LLM provider at a time. There is no mechanism to route different task types (curriculum, action generation, critic, chat) to different models or providers for cost optimization.

THEME 3 — THE DUAL STORE BUG

Two Stores, Same Names, Different Instances

This is perhaps the most insidious bug in the codebase. Two files in the web frontend export identically-named Zustand stores:

// web/src/lib/store.ts:202
export const useControlStore = create<ControlStore>(...)

// web/src/lib/controlStores.ts:26
export const useControlStore = create<ControlStore>(...)

These are different Zustand store instances with different state. The SocketProvider.tsx component (which handles all real-time WebSocket events) imports from @/lib/controlStores (line 14). Every UI component imports from @/lib/store.

Result: the SocketProvider correctly receives socket events and writes command updates, mission updates, marker changes, zone changes, route changes, squad changes, and role changes — into a store that nothing reads. The UI reads from a parallel store that never receives socket data. Real-time updates across the entire control platform are silently dead.

THEME 4 — FRONTEND-BACKEND TYPE MISMATCHES

TYPE-001CRITICAL

Marker/Zone/Route Types Incompatible

Backend MarkerStore.ts uses WorldTypes.ts interfaces (Marker, Zone, Route) with fields like createdAt: number (epoch). Frontend store.ts defines its own Marker/Zone/Route types with different field names and shapes. Even if the API worked, the frontend could not render the data without transformation.

src/control/WorldTypes.ts • web/src/lib/store.ts
TYPE-002HIGH

Command Status Enum Mismatch

Backend CommandTypes.ts uses statuses: 'queued' | 'started' | 'succeeded' | 'failed' | 'cancelled'. Frontend controlStores.ts:15 mirrors this correctly, but store.ts uses 'pending' | 'running' | 'completed' | 'failed'. Status filtering, badge colors, and progress indicators all break.

src/control/CommandTypes.ts • web/src/lib/store.ts • web/src/lib/controlStores.ts
TYPE-003HIGH

Metrics Endpoint Reads Wrong Fields

The metrics/stats pages query for field names that do not match the backend's actual response shape. Counters always display zero because the frontend destructures properties that do not exist on the response object.

web/src/app/metrics/page.tsx • web/src/app/stats/page.tsx
TYPE-004HIGH

Socket Payload Shape Mismatch

Squad-related socket events emit arrays from the backend, but the SocketProvider handler expects an object with an id field for upsert logic. The handler silently fails to update state because Array.find() is called on what was supposed to be an object.

web/src/components/SocketProvider.tsx • src/control/SquadManager.ts

THEME 5 — BOT BEHAVIOR BUGS

Event Listener Leaks

Several action files register event listeners without guaranteed cleanup. placeBlock.ts:49 registers a goal_reached listener via bot.once() but if the timeout fires first, the listener is never removed. Over hundreds of actions, this creates listener accumulation.

attack.ts:80-81 registers listeners for entityGone and itemDrop events cast through as any. These are non-standard mineflayer events that may never fire, leaving the listeners attached indefinitely.

src/actions/placeBlock.ts:49 • src/actions/attack.ts:56-81
Container Operations Hang Forever

All four openContainer calls in container.ts (lines 59, 89, 111, 196) await the container open with no timeout wrapper. If the Minecraft server does not respond (chunk unloaded, server lag, block destroyed mid-open), the promise hangs indefinitely, blocking the bot's entire action loop.

Only walkTo.ts:28 handles the noPath status from the pathfinder. All other movement code in actions and the control platform ignores pathfinding failures entirely. Bots silently fail to move and report success.

src/actions/container.ts:59,89,111,196 • src/actions/walkTo.ts:28
Build System Requires OP

The BuildCoordinator.ts:676 places blocks via /setblock chat commands, which require server operator privileges. The survival-mode buildSchematic.ts tries real block placement first but falls back to /setblock on any error (lines 93-127). CodeExecutor.ts:249 also uses /setblock directly.

On any server where bots lack OP, the entire building system is non-functional. The survival-mode fallback means errors in legitimate placement silently degrade to cheating rather than reporting failure.

src/build/BuildCoordinator.ts:676 • src/actions/buildSchematic.ts:93-127 • src/voyager/CodeExecutor.ts:249

THEME 6 — SOCIAL SYSTEM: EXCELLENT DESIGN, ZERO WIRING

A Complete Social Architecture That Nobody Calls

The social system is beautifully designed. SocialMemory.ts implements memory storage with decay, importance weighting, emotional state tracking (mood, energy, social satisfaction), and a reflection system. BotComms.ts provides inter-bot messaging with message types and unread queues. AffinityManager.ts tracks player-bot relationships with bonuses and decay.

The personality prompt system at src/ai/prompts/personality.ts has full support for socialContext — including nearby bots, memory context, emotional state, and relationship summaries (lines 17-42). The code to generate rich, personality-aware, socially-contextual LLM prompts is complete.

But none of it is called:

BotManager.ts:41 does create a new BotComms() instance and stores it. BotManager.ts:161 exposes getBotComms(). But no BotInstance ever calls sendMessage() or polls getUnread(). The messaging system exists but bots never talk to each other.

Affinities Stuck at 50, Personality Is Cosmetic

The AffinityManager implements a sophisticated relationship system with bonuses for positive interactions and decay over time. However, the NEGATIVE_WORDS array is empty (chat.ts:2), so negative sentiment is never detected. Affinity decay outpaces the rate of bonuses from normal interactions, causing all relationships to converge toward the neutral midpoint of 50.

Combat hits do not reduce affinity because the hit detection code never calls the affinity adjustment methods. Player deaths near bots are not tracked. The relationship system has no functioning negative signal.

Personality: Task Pools Only

Personality affects one thing: the 65% chance of picking a personality-weighted task in CurriculumAgent.ts:372. Beyond that, all six personality types (merchant, guard, explorer, farmer, blacksmith, elder) share identical action libraries, identical movement code, identical combat behavior, and identical inventory management. The personality systemPromptFragment only changes chat flavor text.

THEME 7 — PERSISTENCE ISSUES

Shutdown Doesn't Flush Debounced Writes

Multiple managers use debounced saves (2-second delay via SAVE_DEBOUNCE_MS). The shutdown handler at src/index.ts:149-159 calls eventLog.shutdown() and botManager.removeAllBots(), then process.exit(0). It does NOT call shutdown() on CommanderService, MarkerStore, RoleManager, SquadManager, RoutineManager, BlockerMemory, or BlackboardManager.

Any data written in the last 2 seconds before shutdown is lost. The BotManager.shutdownPersistence() method at line 119 flushes AffinityManager, SocialMemory, and BlackboardManager, but the shutdown handler does not call it either.

src/index.ts:149-162 • src/bot/BotManager.ts:119-122
Memory-Only Systems Lost on Restart

ConversationManager stores all conversation history in memory with no disk persistence. Every bot restart loses all conversation context. BotComms stores messages in memory only — unread messages vanish on restart (though since sendMessage is never called, this is academic).

The CommanderService metrics (total parses, success rate, confidence averages) are "intentionally ephemeral" per the code comment at line 100. But the UI metrics pages expect historical data that will never survive a restart.

src/personality/ConversationManager.ts • src/social/BotComms.ts • src/control/CommanderService.ts:100-111
Synchronous Blocking Writes

Multiple critical paths use fs.writeFileSync: CommandCenter.ts:621, MissionManager.ts:550, BotManager.ts:231, CurriculumAgent.ts:531-534 (four files written synchronously in sequence), and ChainCoordinator.ts:145.

Some managers use atomic temp-file-then-rename patterns (AffinityManager, SocialMemory, BlackboardManager, BlockerMemory). Others write directly to the target file. A crash mid-write on the direct writers risks corruption of commands, missions, bot configs, completed tasks, and skill data.

Multiple files across src/control/, src/voyager/, src/bot/

THEME 8 — BUILD & SUPPLY CHAIN

BuildCoordinator: /setblock Only, No Verification

BuildCoordinator.ts:673-676 places every block via /setblock chat command. There is no post-placement verification that the block actually appeared. The coordinator tracks build progress optimistically — marking blocks as placed the moment the command is sent.

Meanwhile, buildSchematic.ts in src/actions/ contains a proper survival-mode implementation that attempts real block placement with bot.placeBlock(). This code is functional but is not used by the BuildCoordinator. The coordinator and the action exist in parallel with no connection.

Task completion throughout the Voyager pipeline uses fragile substring matching to determine if a task succeeded. The critic agent evaluates based on text parsing of execution output rather than verifiable world state checks.

Supply Chain: Hope-Based Item Handoff

The ChainCoordinator defines supply chain steps as text descriptions that get fed to the LLM as task prompts. There is no actual item handoff mechanism — no giveItem coordination between bots, no shared chest system, no verification that Bot A actually produced the items Bot B needs.

The chain "works" by hoping that when Bot A is told "mine 10 iron ore" and Bot B is told "smelt iron ore into iron ingots," Bot B will somehow acquire Bot A's output. In practice, each bot operates independently with no inventory sharing, making multi-step supply chains impossible.

The ChainCoordinator persists chain definitions to data/supply_chains.json via synchronous writes (ChainCoordinator.ts:145) but is never instantiated in production.

src/supplychain/ChainCoordinator.ts • src/build/BuildCoordinator.ts:673-676

THEME 9 — MAP SYSTEM

550+ Lines of Dead Drawing Code

The map page at web/src/app/map/page.tsx (631 lines) contains extensive canvas drawing code for zone overlays, route visualization, marker rendering, and terrain coloring. None of this renders because the backend has no endpoints serving terrain data, zone boundaries, or route waypoints.

The frontend zone and marker data models are completely incompatible with the backend WorldTypes.ts definitions (see Theme 4). Even if endpoints existed, the data would not map correctly to the drawing code's expected input shapes.

Currently, the map functions as a dot tracker: it shows bot positions (which do work through the correct store) on a blank canvas. All zone highlighting, route drawing, and marker overlay code is present but unreachable.

What Could Be

The map system's design is ambitious and well-structured. Zone drawing supports rectangular and circular areas with color coding by type (guard, farm, build). Route visualization draws connected waypoint sequences. Marker rendering handles named positions with icons.

The gap is entirely in the backend: MarkerStore has full CRUD for markers, zones, and routes but is never instantiated (Theme 1). No API routes serve this data to the frontend. And the data models on each side use different field names and structures (Theme 4).

Wiring this up requires: (1) instantiate MarkerStore in production, (2) connect API routes to it, (3) fix type mismatches between FE/BE, (4) fix the dual store bug so socket updates flow to the UI. Four fixes, each individually small, that would bring the entire map system to life.

THEME 10 — DOCUMENTATION & TESTING

CLAUDE.md: ~80% Fictional API Documentation

The project's CLAUDE.md lists 30+ API endpoints under "Control Platform API" including full CRUD for commands, missions, markers, zones, routes, squads, roles, and commander operations. It describes lifecycle actions (start, pause, resume, cancel, retry) for missions, squad membership management, role assignment CRUD, and natural language command parsing.

Roughly 80% of these endpoints either do not exist as routes, or route to services that are never instantiated. The documentation reads as a specification for what the system should do, not what it actually does. For a file that AI tools read to understand the codebase, this is actively harmful — it causes generated code to call endpoints that 404.

Config options that do nothing: maxConcurrentRequests is defined in config.yml:61 and typed in config.ts:56 but never referenced by any LLM client code. No rate limiting is enforced regardless of this setting.

Testing: Only Mocked Control Tests

Seven test files exist, all in test/control/: CommandCenter, CommanderService, MarkerStore, MissionManager, RoleManager, SquadManager, and an integration test. These are well-written but test only mocked in-memory behavior of the very managers that are never instantiated in production.

There is no test script in the root package.json. Zero API route tests. Zero tests for BotInstance, VoyagerLoop, CodeExecutor, any LLM client, or any action primitive. The three most complex runtime files have zero coverage:
